## Vision

### 10x Check
The 10x version of Date Drop isn't a dating app — it's the **campus social graph for romance**. Every student on campus is either on the app, has been nominated by a friend, or has received an invite. The Cupid feature creates a social layer where friends actively help each other find partners. The invite loop makes growth automatic — each user brings 2-3 others. Weekly match cycles create ritual and anticipation ("Match Monday"). Within one semester, Date Drop is *the* way students meet romantic partners at a university, displacing confession walls entirely.

The key insight: dating apps fail when they're transactional ("swipe, match, chat"). Date Drop succeeds by being **social** — friends nominate friends, invite links create curiosity, and the community grows through trust networks rather than cold acquisition.

## Accepted Scope (added to this plan)

### Invite Link System
- **Data model:** New `invites` table — `id`, `inviter_id` (FK to profiles), `invite_code` (unique 8-char alphanumeric), `used_by` (nullable FK), `created_at`, `expires_at` (30 days from creation)
- **URL format:** `/invite/<code>` — lands on campus-specific page (see below)
- **Quota:** All users get 5 invite links. Seed users are not special — everyone gets the same quota to keep the system simple.
- **Recipient experience:** Landing page shows mystery: "Someone at [University] thinks you should try Date Drop. [X]+ students have already joined. Sign up to find out who." Inviter identity is NOT revealed on the landing page (curiosity drives conversion). After signup and questionnaire completion, the inviter's name is revealed on the home page as a reward for completing the flow.
- **Quota enforcement:** `SELECT COUNT(*) FROM invites WHERE inviter_id = :uid`; reject creation if >= 5.
- **Code generation:** `secrets.token_urlsafe(6)` (yields 8 base64url chars). Case-sensitive.
- **Edge cases:** Expired links show "This invite has expired." Non-whitelisted school emails are rejected at signup (existing behavior). Duplicate signups from the same invite are prevented by `used_by` uniqueness. Self-invites blocked (inviter_id != signed-up user). Each invite code can only be used once. If visitor is already authenticated, redirect to `/home` with a toast: "You already have an account."

### Email Notifications
- **Provider:** Resend free tier (3,000 emails/month — sufficient for launch). Integration via REST API with API key. If approaching the 3,000 limit, upgrade to Resend paid tier ($20/mo for 50k emails). Monitor usage via Resend dashboard.
- **Trigger:** Email is sent from the Flask route that inserts matches (manual SQL insert or `POST /api/admin/generate-matches`), not a database trigger. No Supabase edge functions needed.
- **Content:** "You have a new match on Date Drop! Open the app to see who: [link]"

### Questionnaire Analytics
- **Tracking:** Log `questionnaire_started` and `questionnaire_completed` events with timestamps and the last completed dimension. Store in a simple `analytics_events` table (columns: `id`, `user_id`, `event_type`, `metadata_json`, `created_at`).
- **Drop-off:** Track which dimension was last completed before abandonment.

### Quick Match Mode
- **Dimensions (6 of 16):** `s_introversion`, `s_ambition`, `s_romance_style`, `s_social_energy`, `s_lifestyle`, `s_values` — these are the dimensions most likely to matter for initial compatibility based on dating research.
- **Partial matching:** When matching a quick-match user against a full-profile user, score only on shared dimensions and normalize the score to a 0-100 scale. Example: if both completed 6 dimensions, score across 6; if one has 6 and the other has 16, score across the 6 shared dimensions.
- **UX:** Toggle shown at the top of the questionnaire page before the user begins answering: "Get matched fast (2 min)" vs "Complete profile (5 min)". Quick match users see a prompt to complete their full profile after receiving their first match.
- **Timing:** Partial matching normalization is built in Phase 1 but only activated when algorithmic matching begins in Phase 4. During manual matching (Phases 2-3), the founder can see which dimensions each user completed in the Supabase dashboard.

### Campus Landing Page
- **Social proof counter:** Use thresholds, not exact counts — "10+ students have joined" / "50+ students" / "100+ students." Only show the counter once the threshold of 10 is reached. Below 10, show "Be one of the first at [University]."
- **Privacy:** No individual names or identifying information shown.

## Revised Strategy: Cupid-Parallel Launch

The original design doc recommended a 20-30 person manual pilot. The CEO review challenged this: **critical mass is the real risk**, not match quality. The revised strategy:

### Phase 1: Build the Growth Engine (CC: ~2 hours)
1. Build invite link feature — new route, `invites` table, invite landing page (~30 min)
2. Build email notification system — Resend integration, match trigger (~20 min)
3. Build campus landing page — threshold-based counter, school-specific branding (~15 min)
4. Add questionnaire analytics — event tracking, drop-off logging (~15 min)
5. Add quick match mode — 6-dimension flow, partial matching logic (~30 min)

### Phase 2: Seed the Network (Founder: ~3-5 days)
1. Recruit 15-20 seed users from personal network (more seeds needed — see growth math below)
2. Each seed user gets 5 invite links to share
3. **Growth math (realistic):** 20 seeds x 5 invites x 20% conversion = 20 first-degree signups. All new signups also receive 5 invite links, enabling second-degree growth. Those 20 x 5 invites x 15% conversion = 15 second-degree. Total week 1 target: **50-55 signups** (not 100). If conversion exceeds 20%, growth compounds faster.
4. Founder manually matches the first 2-3 batches using Supabase Table Editor directly — export profiles, match by hand, insert into `matches` table. No engineering work required for manual matching — no admin UI needed for this phase.

### Phase 3: First Match Cycle (Days 7-10)
1. Once 50+ profiles exist, founder makes manual matches
2. Email notifications sent automatically to matched pairs
3. Monitor chat initiation rate and questionnaire completion rate
4. Iterate on match quality based on user feedback

### Phase 4: Automate (After 100+ profiles)
1. Build matching algorithm informed by manual matching learnings
2. Switch from manual to algorithmic matching
3. Cupid nominations boost match scores for nominated pairs
4. Weekly automated match cycles begin

### Kill Criteria (updated)
- If invite links produce < 2x growth (each user brings < 2 new signups): investigate messaging and friction
- If questionnaire completion < 50%: activate quick match mode as default
- If chat initiation < 30% after first match cycle: match quality problem — interview users
